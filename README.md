# 회의록 액션아이템 관리 서비스

회의록 텍스트를 입력하면 AI가 결정사항 / 액션아이템 / 논의사항으로 구조화하고, 그 결과를 사람이 검토·수정·저장하고 회의별로 관리하는 웹서비스입니다.

- 레포: [https://github.com/yeehyun2ya/meeting-action-items](https://github.com/yeehyun2ya/meeting-action-items)
- 배포 데모: [http://43.200.182.174:3000](http://43.200.182.174:3000)

## 1. 무엇을 만들었고 어디까지 됐는가

회의록 원문을 붙여넣으면 LLM이 세 갈래(결정사항·액션아이템·논의사항)로 분리하고, 사용자가 그 결과를 화면에서 검토·수정한 뒤 저장하면 회의별로 조회·수정·삭제할 수 있습니다.

### 구현 완료 범위

- 관계형 데이터 모델: `Meeting` 1 : N (`Decision` / `ActionItem` / `Discussion`) 구조로 설계했습니다. 평평한 단일 테이블이 아니라 회의 1건에 세 종류의 자식 레코드가 붙는 관계이며, 회의 삭제 시 관련 항목은 cascade delete됩니다.
- AI 구조화: 회의록 텍스트를 LLM으로 결정사항, 액션아이템, 논의사항으로 분리합니다. 출력은 JSON 고정 형식으로 강제하고 Zod로 검증합니다.
- AI 호출의 비동기 처리: 프론트는 AI 구조화 요청 중 loading 상태와 스피너를 보여주고, 백엔드는 OpenRouter 호출을 `async/await`로 처리합니다. 30초 타임아웃을 두고, 타임아웃 / 네트워크 오류 / OpenRouter 비정상 응답 / JSON 파싱 실패 / Zod 검증 실패를 구분합니다.
- CRUD: 회의록·액션아이템·결정사항·논의사항 저장/조회/수정/삭제를 구현했습니다. AI 결과는 저장 전 반드시 사람이 검토·수정할 수 있고, 그대로 신뢰해 저장하지 않습니다.
- 프론트-백 분리: Next.js 프론트(3000)와 Express REST API(4000)를 별도 서버로 분리했고, 실제 HTTP 통신으로 PostgreSQL에 저장합니다. 회의록 입력부터 AI 구조화 로딩, 결과 검토·수정, 목록·상세, 항목별 인라인 편집까지 화면으로 구현했습니다(상세는 2번).
- AWS 배포: EC2에 프론트·백엔드, RDS에 PostgreSQL을 구성했습니다. URL로 접속해 동작을 확인할 수 있습니다.

### 미구현 범위 및 후속 개선

- 액션아이템 전역 목록·필터: 현재 액션아이템은 각 회의 상세 화면 안에서 관리됩니다. 여러 회의를 가로지르는 통합 목록(담당자별·기한별·상태별)은 만들지 못했습니다.
- 권장 기능인 서버사이드 필터·정렬, 완료 체크, 누적 대시보드는 미구현입니다.
- 로그인 / 사용자 권한은 미구현입니다. 현재는 인증 없이 단일 사용자 기준으로 동작합니다.
- 파일 업로드는 미구현입니다. 입력은 텍스트 붙여넣기만 지원합니다.
- 운영 수준 배포 보강은 미완입니다. nginx 리버스 프록시, HTTPS, 도메인 연결은 후속 개선으로 남겼습니다.

판단 기준은 명세의 "필수 1~6을 끝까지 연결하는 것이 권장·보너스를 얕게 여러 개 건드리는 것보다 높게 평가된다"는 문장이었습니다. 그래서 필수 플로우를 먼저 완주시키고 권장 기능은 의식적으로 미뤘습니다.

## 2. 아키텍처 설명

### 전체 연결

```text
[브라우저]
    │  HTTP
    ▼
[Next.js 화면 : EC2:3000]
    │  브라우저 fetch / HTTP
    ▼
[Express REST API : EC2:4000]
    │
    ├─ Prisma ──▶ [RDS PostgreSQL]
    │
    └─ HTTPS ──▶ [OpenRouter LLM]
```

프론트와 백엔드는 별도 프로세스로 뜨고, 실제 API 호출은 Next.js client component가 실행되는 브라우저에서 백엔드 API로 직접 보냅니다. 백엔드는 Prisma를 통해 RDS PostgreSQL에 접근하고, 구조화 요청 시에만 OpenRouter LLM을 호출합니다.

CORS는 배포 환경에서 `CORS_ORIGIN` 환경변수로 허용 origin을 제한했습니다. 다만 환경변수가 없으면 기본 동작으로 넓게 열리는 구조라, 환경변수 부재 시 자동 차단(fail-closed)은 후속 개선점입니다.

### 프론트엔드 구현

프론트엔드는 Next.js App Router 기반으로 구현했고, 사용자가 회의록을 입력한 뒤 AI 구조화 결과를 검토·수정·저장하는 흐름을 중심으로 구성했습니다.

가장 중요한 화면은 회의록 입력 → AI 구조화 → 검토/수정 → 저장으로 이어지는 작성 화면입니다. 사용자가 제목과 회의록 원문을 입력하고 `AI로 구조화하기` 버튼을 누르면 프론트는 loading 상태로 전환해 "AI가 회의록을 분석하고 있습니다" 스피너를 보여줍니다. AI 응답이 성공하면 결정사항, 액션아이템, 논의사항을 각각 검토 섹션으로 나누어 보여주고, 사용자가 저장 전에 내용을 직접 수정할 수 있게 했습니다.

AI 결과는 최종 데이터가 아니라 사람이 검토할 초안으로 취급했습니다. 그래서 구조화 직후 바로 저장하지 않고, 검토 화면에서 결정사항·액션아이템·논의사항을 확인한 뒤 `회의 저장` 버튼을 눌러야 DB에 저장됩니다. 이 흐름은 백엔드의 `POST /ai/structure-meeting`과 `POST /meetings` 분리와 맞물려, AI 결과가 사람의 확인 없이 저장되지 않도록 합니다.

저장된 회의는 목록 화면과 상세 화면에서 확인할 수 있습니다. 목록 화면에서는 회의별 결정사항·액션아이템·논의사항 개수와 일부 미리보기를 보여주고, 상세 화면에서는 회의 제목과 각 항목을 인라인으로 수정·삭제할 수 있게 했습니다.

에러 처리도 프론트에서 공통화했습니다. 백엔드의 모든 실패 응답을 `{ code, message, retryable }` 형식으로 통일했기 때문에, 프론트 API 클라이언트는 이를 `ApiClientError`로 정규화합니다. 화면에서는 `ErrorBox`가 `message`와 `code`를 보여주고, `retryable: true`인 경우에만 "다시 시도" 버튼을 노출합니다.

### 데이터 모델 관계도

```text
Meeting (회의)
  ├─ 1:N ─ Decision         (결정사항)
  ├─ 1:N ─ ActionItem       (액션아이템: content, sourceQuote, assignee?, dueDate?, dueDateRaw?)
  └─ 1:N ─ Discussion       (논의사항)
```

- 세 자식 테이블 모두 `meetingId` FK를 갖고, 회의 삭제 시 cascade delete됩니다.
- `ActionItem.assignee` / `dueDate`는 nullable입니다.
- 모든 추출 항목에는 `sourceQuote`가 필요합니다.

"논의사항" 항목은 계층마다 이름이 다릅니다. 계층별 관례를 따른 결과이며, 모두 같은 개념을 가리킵니다.

| 계층 | 이름 | 규칙 |
| --- | --- | --- |
| 화면 표시 | 논의사항 | 사용자 표시용 한국어 |
| DB 모델 (Prisma) | `Discussion` | 단수형 모델명 |
| API 리소스 | `discussion-points` | REST 관례(복수·kebab-case) |
| 프론트 타입 | `DiscussionPoint` | 도메인 의미를 드러낸 이름 |

API 경로가 `/decisions`, `/action-items`, `/discussion-points`처럼 복수·kebab-case인 것은 REST 리소스 관례를 따른 것입니다. 다만 DB 모델명(`Discussion`)과 프론트 타입명(`DiscussionPoint`)이 완전히 일치하지 않는데, 이는 하나로 통일하는 것이 개선점입니다.

세 종류를 단일 테이블에 `type` 컬럼으로 눕히지 않고 별도 테이블 3개로 나눈 이유는 두 가지입니다. 첫째, 한 테이블에 회의 본문과 항목을 같이 두면 회의록 제목·원문이 항목 수만큼 중복됩니다. 둘째, 액션아이템만 담당자·기한·상태를 갖고 결정사항·논의사항은 그렇지 않아, 한 테이블로 몰면 대부분 컬럼이 특정 타입에서만 의미 있는 nullable로 남아 모델 의도가 흐려집니다.

### API 구성

| Method | Path | 역할 |
| --- | --- | --- |
| `GET` | `/health` | 서버 상태 확인 |
| `POST` | `/ai/structure-meeting` | 회의록 텍스트를 구조화 결과로 변환합니다. 저장은 하지 않습니다. |
| `POST` | `/meetings` | 사용자가 검토·수정한 회의 데이터를 저장합니다. |
| `GET` | `/meetings` | 회의 목록을 조회합니다. |
| `GET` | `/meetings/:id` | 회의 상세를 조회합니다. |
| `PATCH` | `/meetings/:id` | 회의 정보를 수정합니다. |
| `DELETE` | `/meetings/:id` | 회의를 삭제합니다. 연결된 항목도 함께 삭제됩니다. |
| `PATCH` / `DELETE` | `/action-items/:id` | 액션아이템을 수정 / 삭제합니다. |
| `PATCH` / `DELETE` | `/decisions/:id` | 결정사항을 수정 / 삭제합니다. |
| `PATCH` / `DELETE` | `/discussion-points/:id` | 논의사항을 수정 / 삭제합니다. |

항목(액션아이템·결정사항·논의사항)은 회의와 별개로 항목 단위 엔드포인트를 두어, 상세 화면의 인라인 편집에서 개별 항목만 부분 수정·삭제할 수 있게 했습니다.

## 3. 내가 내린 주요 결정과 그 이유

### AI 구조화 API와 저장 API 분리

`POST /ai/structure-meeting`은 구조화 결과만 돌려주고 저장하지 않습니다. 저장은 사용자가 검토·수정한 뒤 `POST /meetings`로 따로 합니다.

명세의 "AI 결과를 그대로 신뢰해 저장 금지"를 API 경계 자체로 강제한 것입니다. 한 엔드포인트에서 구조화와 저장을 동시에 하면 사람의 검토 단계를 코드가 건너뛸 수 있어, 이를 구조적으로 막았습니다.

### 프론트 상태 흐름을 input / loading / review로 분리

AI 구조화는 시간이 걸리고 실패할 수 있는 작업이기 때문에, 입력 화면과 검토 화면 사이에 loading 상태를 명확히 두었습니다. 사용자가 구조화 버튼을 누르면 입력 폼을 잠그고 스피너를 보여주며, 성공하면 review 화면으로 전환하고 실패하면 다시 input 화면으로 돌아와 에러를 보여줍니다.

이렇게 한 이유는 AI 결과를 바로 저장하지 않고 사람이 검토하는 흐름을 화면에서도 강제하기 위해서입니다. API는 `POST /ai/structure-meeting`과 `POST /meetings`로 분리했고, 프론트도 입력 → 로딩 → 검토 → 저장 단계로 나누어 같은 의도를 유지했습니다.

### AI 결과를 검증 가능한 초안으로 다루는 결정

AI 구조화 결과는 최종 데이터가 아니라 사람이 검토할 초안으로 취급했습니다. 원문에 없는 담당자·기한을 AI가 채우지 않도록 nullable 필드를 허용하고, 모든 항목에 `sourceQuote`를 두어 원문 근거를 확인할 수 있게 했습니다. 구체적인 검증 방식은 4번에서 설명합니다.

### 삭제 정책: RESTRICT에서 Cascade로 변경

처음엔 Prisma 기본 동작인 RESTRICT로 테이블을 만들었는데, 삭제 API를 설계하면서 회의가 삭제되면 그 회의의 결정사항·액션아이템·논의사항은 존재 의미가 없다고 판단해 `onDelete: Cascade`로 바꿨습니다. 자식이 부모 없이 남는 orphan을 남기지 않기 위한 선택입니다.

### 실패 응답의 retryable을 서버가 판단

실패 원인을 가장 정확히 아는 서버가 재시도 가능 여부를 판단하도록 했습니다. 이 결정은 프론트 에러 처리를 단순하게 만들기 위한 것으로, 구체적인 응답 형식과 UI 반영은 5번에서 설명합니다.

### 스택 선택

- Next.js(App Router) + TypeScript + Tailwind: 프론트 화면 구현 경험이 있는 스택이라 짧은 기간에 UI를 빠르게 세우기 좋았습니다.
- Express + TypeScript: 프론트와 타입을 공유해 API 응답 형태를 일관되게 맞추기 위해 백엔드도 TypeScript로 통일했습니다.
- PostgreSQL + Prisma: 관계형 모델(1:N, cascade)을 스키마로 명시적으로 표현하고 마이그레이션으로 관리하기 위해 선택했습니다.
- Zod: LLM 출력을 타입으로 신뢰하지 않고 런타임에서 형식을 검증하기 위해 사용했습니다.
- OpenRouter `google/gemma-4-26b-a4b-it:free`: 과제 요건에 맞춰 무료 티어 모델을 사용했습니다.
- AWS EC2 + RDS: 과제 요건에 따라 애플리케이션과 DB를 분리해 배포했습니다.

### 작업 순서 판단

배포까지 얇게라도 한 줄로 관통시키는 것을 먼저 목표로 잡고, 권장·보너스 기능은 필수 완주 뒤로 미뤘습니다. 데이터 모델 → 저장/조회 API → AI 구조화 → 실패 처리 → 프론트 플로우 → 배포 순서로 진행했습니다.

## 4. AI 구조화의 신뢰성을 어떻게 확보했는가

hallucination은 한 겹으로 막기 어렵다고 봐서, 형식 강제·nullable·원문 대조·책임 분리를 겹쳐 걸었습니다.

### 1) 출력 형식 강제 + 런타임 검증

시스템 프롬프트에서 결정사항·액션아이템·논의사항 세 카테고리의 JSON 스키마를 고정하고, "원문에 없으면 지어내지 말고 null로 둘 것"을 명시했습니다. 그리고 LLM 응답을 Zod `.strict()`로 파싱·검증해, 스키마에 정의하지 않은 필드가 오거나 형식이 어긋나면 저장 흐름에 태우지 않습니다.

### 2) nullable = 지어내기 방지 장치

`assignee`와 `dueDate`를 nullable로 둔 것은 단순한 스키마 편의가 아니라 hallucination 방지 설계입니다. 원문에 담당자나 기한이 명시돼 있지 않으면 AI가 그럴듯한 값을 채우는 대신 `null`을 반환하고, 사람이 검토 단계에서 직접 채웁니다.

### 3) sourceQuote 필수 + 원문 포함 여부 검증

추출된 모든 항목은 원문의 어느 문장에서 나왔는지 `sourceQuote`를 필수로 반환하도록 강제했습니다. 여기서 한 걸음 더 나아가, Zod로 문자열 존재만 확인하는 데 그치지 않고 서버에서 그 인용문이 실제 회의록 원문에 포함돼 있는지 검증합니다. 공백 정규화 후 포함 여부를 확인하고, 원문에 없는 근거면 AI 응답을 실패로 처리합니다.

### 4) 상대 날짜를 LLM이 계산하지 않게 분리

"다음 주까지" 같은 상대 표현을 LLM이 실제 날짜로 계산하게 두면, 원문에 근거 없는 날짜를 만들어낼 수 있습니다. 그래서 LLM에는 원문 표현 추출(`dueDateRaw`)까지만 맡기고, 원문에 명시적 날짜가 없으면 `dueDate`는 `null`로 두도록 했습니다.

현재 구현은 `dueDateRaw`로 원문 표현을 보존하고 `dueDate`를 지어내지 않게 막는 단계까지입니다. 상대 날짜를 실제 날짜로 변환하는 서버 규칙은 아직 구현하지 않았고, 후속 개선에서 결정론적 로직으로 처리할 계획입니다.

### 5) 사람 검토 단계

AI 결과는 최종본이 아니라 사람이 고칠 초안입니다. 상세 화면은 인라인 편집으로, 항목별 엔드포인트(`PATCH /action-items/:id` 등)를 통해 개별 항목만 부분 수정할 수 있습니다.

### RAG를 쓰지 않은 이유

검색 증강(RAG)은 정답이 외부 지식에 있을 때 유효합니다. 이 과제는 정답이 입력된 회의록 텍스트 안에 이미 존재하므로, 외부에서 끌어올 것이 없습니다. 오히려 RAG를 얹으면 원문에 없는 내용이 섞여 들어올 여지가 생겨 hallucination 방지 목적에 역행합니다. 그래서 의도적으로 배제했습니다.

## 5. 막혔던 지점과 어떻게 풀었는지

구현 과정에서 가장 많이 보강한 부분은 API 경계와 AI 호출 실패 처리였습니다. AWS 배포는 EC2에 프론트와 백엔드를 올리고 RDS PostgreSQL을 연결하는 단순한 구조로 진행해 비교적 수월했지만, AI 구조화 결과를 안전하게 다루기 위해 입력 검증과 에러 응답 계약을 여러 차례 다듬었습니다.

### 1) API 입력 검증 보강

검증 과정에서 깨진 JSON, 필수 값 누락, 실제로 존재하지 않는 날짜가 서버 경계를 통과할 수 있는 부분을 발견했습니다. 이를 막기 위해 요청 본문이 올바른 JSON이 아니면 `400 INVALID_JSON`으로 응답하고, 필수 값 누락이나 잘못된 날짜는 `400 VALIDATION_ERROR`로 거부했습니다.

특히 날짜는 ISO 형식 여부만 확인하지 않고, `Date`로 파싱한 뒤 다시 `toISOString()` 한 값이 원본과 같은지 비교했습니다. 이를 통해 `2026-02-31T00:00:00.000Z`처럼 형식은 그럴듯하지만 실제로 존재하지 않는 날짜가 저장되지 않도록 했습니다.

### 2) AI 호출 실패 원인 분리

AI 구조화 API는 OpenRouter라는 외부 서비스를 호출하기 때문에 실패 원인이 다양했습니다. 처음에는 실패를 단순히 하나의 오류로 처리할 수 있었지만, 검증 과정에서 타임아웃, 네트워크 오류, OpenRouter 실패 응답, AI 응답 JSON 파싱 실패, Zod 검증 실패를 구분할 필요가 있다고 판단했습니다.

그래서 OpenRouter 호출에 30초 타임아웃을 두고, 실패 원인별로 응답을 나누었습니다. 타임아웃은 `504 OPENROUTER_TIMEOUT`, 네트워크 오류는 `502 OPENROUTER_NETWORK`, OpenRouter의 비정상 응답은 `502 OPENROUTER_REQUEST_FAILED`, AI 응답 구조가 맞지 않는 경우는 `502 AI_RESPONSE_INVALID`로 처리했습니다.

### 3) 에러 응답 형식 통일

프론트에서 에러를 일관되게 처리하려면 API마다 다른 형태로 에러를 내려주면 안 된다고 판단했습니다. 그래서 모든 실패 응답을 `{ code, message, retryable }` 형식으로 통일했습니다.

`code`는 프론트가 에러 종류를 구분하기 위한 값이고, `message`는 사용자에게 보여줄 수 있는 설명이며, `retryable`은 다시 시도하면 해결될 가능성이 있는지를 서버가 판단한 값입니다. 예를 들어 AI 호출 타임아웃이나 네트워크 오류는 `retryable: true`로 두고, 잘못된 요청 형식이나 AI 응답 검증 실패는 `retryable: false`로 처리했습니다.

프론트는 이 값을 보고 재시도 가능한 오류일 때만 "다시 시도" 버튼을 보여줄 수 있습니다.

### 4) EC2 메모리 제약

AWS 배포 자체는 비교적 빠르게 완료했지만, EC2 `t3.micro` 환경에서 Next.js 빌드 중 메모리 부족이 발생했습니다. 인스턴스 크기를 키우기보다 과제 데모 범위에서는 2GB 스왑을 추가해 빌드를 완료했습니다.

이 과정에서 로컬에서 정상 동작하는 애플리케이션도 배포 환경의 리소스 제약에서는 별도 대응이 필요하다는 점을 확인했습니다.

### 5) 배포 범위 트레이드오프

배포는 프론트(3000)와 백엔드(4000)를 EC2에서 직접 노출하는 방식으로 완료했습니다. 과제 데모에서는 실제 접속 가능한 동작 확인을 우선했기 때문에 nginx 리버스 프록시, HTTPS, 도메인 연결은 적용하지 않았습니다.

운영 수준으로 확장한다면 nginx를 통해 단일 진입점으로 묶고, HTTPS와 도메인 연결을 적용하는 것이 우선 개선점입니다. RDS는 외부에 직접 열지 않고 EC2 보안그룹에서만 접근하도록 제한했습니다.

## 6. AI 개발 도구(바이브코딩)를 어디에 어떻게 썼는지 + 시간이 더 있었다면

### AI 개발 도구를 사용한 방식

코드 생성은 AI 코딩 도구를 사용하되, 스키마 설계·API 경계·작업 순서 같은 판단은 직접 내렸습니다. 구조화 결과가 세 카테고리를 갖도록 강제하는 것, structure/save를 분리하는 것, nullable과 `sourceQuote` 원문 검증을 hallucination 방지로 설계하는 것 등은 제가 정하고 도구에 지시했습니다.

작업 중 도구가 만든 결과를 그대로 믿지 않고, 커밋 전에 스펙과 실제 동작을 대조했습니다. 특히 API 검증과 AI 실패 처리는 검증 과정에서 문제를 발견한 뒤 별도 fix 커밋으로 보강했습니다.

### 시간이 더 있었다면

- STT 날것 입력의 필러 전처리: 실제 회의 녹취 텍스트에는 "아, 음, 네 좋구요" 같은 필러가 섞입니다. 입력 정규화 단계를 두면 구조화 품질이 더 안정적일 것입니다.
- `sourceQuote` 원문 검증 정교화: 현재 검증은 공백 정규화 후 포함 여부 기반입니다. 어절·문장 단위 근거 검증으로 더 정교화할 수 있습니다.
- CORS fail-closed: 환경변수 부재 시 넓게 열리는 현재 구조를, 값이 없으면 자동 차단하도록 바꾸는 것이 다음 개선점입니다.
- 미룬 권장 기능: 액션아이템 전역 목록, 서버사이드 필터·정렬, 완료 대시보드, 인증을 이어서 진행할 수 있습니다.
- 운영 배포 보강: nginx 리버스 프록시, HTTPS, 도메인 연결, 프로세스 자동 재시작, 로그 관리를 추가할 것입니다.
